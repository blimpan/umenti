## File overview
This file contains the painpoints we've mapped out from both the teacher's and student's point of view based on how most schools function today.
They are organized into tables. Use this information to better understand what we're doing and why. Note that there can be multiple row entries for the same painpoint.

## Teacher's POV
| Painpoint | Suggested feature | Additional implementation details |
| --- | --- | --- |
| Limited insight into students' comprehension | Derive insights from students' interactions with platform | As the student answers and asks questions, have the platform track in the DB which concepts they have understood, which concepts they haven't understood and the effort put into them. For example, a concept the student has understood is of little interest to the teacher, but a concept which the student has failed 5 questions for and still doesn't understand is of interest. |
| Limited resources to create or update course content | Use LLMs to create content based on teacher's instructions and provided material | When creating a course in the platform from scratch, we need determine the following: (1) Subject, (2) Modules the subject is broken down into, (3) Learning objectives (the overall skill the teacher want to develop), (4) Learning outcomes of each objective (the specific skills that the student will be taught in order to develop the overall skill), (5) The target audience (e.g. kids age 13-15), (6) Language. | In almost all cases (if not all) this information can be taken from official state sources (e.g. the Swedish National Agency of Education) and structured in advance so that the teacher can just select their specific course within their country/state's curriculum. The teacher should be able to either manually update the course's structure and knowledge base (e.g. reviewed theory and exercises) manually themselves or with the help of an AI assistant that can help them iterate on their thoughts , scaffold new content or walk them through how to do it (like which page to go to in order to change the content of a specific module). |
| Can't give everyone personalized feedback during learning process | Use LLMs to respond to students' questions and answers (based on their history) | Addressed in Student's POV |
| Grading & feedback for assignments and exams takes up a majority of working hours outside of the classroom | Use LLMs to do an initial review of submissions according to defined criteria and have the teacher verify it (and correct if necessary). Just like devs do with Cursor. | After deadline for submission has passed, each submission is turned into structured output that can be passed to an LLM along with the grading criteria. The grading assistant generates comments as if it were a teacher (e.g. "this paragraph is poorly structured and doesn't connect with the previous one" or "here the student forgot to add the constant after integrating the function") and a suggested grade with reasoning. The teacher can later use this initial screening as a foundation to more quickly and hopefully more objectively grade the submission. The feedback from the grading assistant that the teacher decides to keep and that they add themselves is later shown to the student and saved to the knowledge base about the student so that it can be referred back to in any future similar assignment and so that the student can be given the task of implementing the feedback. |
| Ends up grading the work of AI instead of student due to outdated examination formats and lazy AI usage by student. [Outside scope for now] | Have the solved exercises in the platform serve as a part of the evaluation |  |
